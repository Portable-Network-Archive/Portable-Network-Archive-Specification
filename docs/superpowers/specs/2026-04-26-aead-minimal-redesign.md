# PNA AEAD Minimal Redesign

**Date:** 2026-04-26
**Status:** Spec design (pre-implementation)
**Supersedes:** Draft commit `a31aa01` on branch `spec/aead-gcm-introduction` (AENC/FENC chunks, archive-level AEAD framework)

## Background

PNA's prior AEAD draft (commit `a31aa01`) introduced two new critical chunk types (`AENC` per-archive, `FENC` per-entry), an 86-byte AAD with metadata SHA-256, and an HKDF-based per-entry key derivation. Brainstorming review on 2026-04-26 concluded that this design was over-engineered for PNA's threat model and design philosophy.

This document specifies a **minimal AEAD design** that achieves equivalent core security guarantees (per-chunk authentication, intra-entry reorder/truncation defense, intra-archive entry reorder defense) with **zero new chunk types**, **no HKDF**, and an **11-byte AAD**.

## Design Principles

1. **Categorize each field** as one of:
   - **A. Algorithm-required** — without it, GCM does not work
   - **B. Security-recommended** — defends a specific tampering class
   - **C. PNA compatibility** — supports format evolution
   Drop any field whose marginal defense is below PNA's threat model.

2. **Reuse existing PNA chunks (PHSF)** rather than introducing new chunk types. PNG-style spec philosophy: critical chunks are added only when no existing chunk can serve the role.

3. **Self-contained per-entry encryption.** Each entry carries everything needed to decrypt its own data. No archive-wide AEAD context state.

4. **Future-version domain separation by Cipher mode value, not by AAD magic.** Each new AEAD framework version MUST allocate a new FHED `Cipher mode` value, ensuring AAD bytes naturally differ across versions.

## Final Design

### 1. AHED `Minor version`

| AHED Major | AHED Minor | Meaning |
|---|---|---|
| 0 | 0 | Baseline (CBC, CTR only) |
| 0 | 1 | AEAD framework introduced. Reserves FHED `Cipher mode` value 2 for GCM. |

A decoder that does not implement Minor 1 MUST report a fatal error if it encounters a `Cipher mode = 2` entry (per the standard "unrecognized Cipher mode value in critical chunk" rule).

### 2. FHED `Cipher mode = 2` (GCM)

| Cipher mode | Mode | AHED Minor required |
|---|---|---|
| 0 | CBC | 0 |
| 1 | CTR | 0 |
| 2 | GCM (AEAD) | 1 |

Combined with `Encryption method`:
- `Encryption method = 1` + `Cipher mode = 2` → AES-256-GCM (AES-GCM, NIST SP 800-38D)
- `Encryption method = 2` + `Cipher mode = 2` → Camellia-256-GCM (RFC 6367)

### 3. PHSF (placement and semantics for AEAD)

PHSF placement is **unchanged** from Minor 0: PHSF appears per-entry, after FHED.

For an AEAD entry (`Cipher mode = 2`):
- The PHSF chunk's password hash output (32 bytes from Argon2id or PBKDF2) is used **directly** as the GCM key. No HKDF.
- Encoders **SHOULD** emit byte-identical PHSF chunks for all AEAD entries within a single archive when those entries share a password. This permits decoders (and encoders) to cache the Argon2id/PBKDF2 result and avoid re-running the password hash per entry.
- Encoders **MAY** emit different PHSF chunks per AEAD entry if entries genuinely use different passwords. Decoders MUST handle both cases.

### 4. Per-FDAT/SDAT chunk layout (when `Cipher mode = 2`)

```
[12 byte random nonce] [ciphertext] [16 byte GCM tag]
```

- The chunk's standard length field counts all three regions: `12 + len(ciphertext) + 16`.
- The nonce occupies the first 12 bytes of the chunk's data field.
- The tag occupies the final 16 bytes.
- Decoders MUST reject any FDAT/SDAT chunk shorter than 28 bytes when the entry uses `Cipher mode = 2`.

The chunk's existing CRC32 covers the whole `(nonce || ciphertext || tag)` concatenation. CRC32 detects transit corruption; the GCM tag is the authoritative authenticity check.

### 5. AAD construction (11 bytes, fixed)

For each FDAT/SDAT chunk:

| Bytes | Field | Source |
|---|---|---|
| 1 | Encryption method | FHED `Encryption method` byte |
| 1 | Cipher mode | FHED `Cipher mode` byte (= 2 for GCM v1) |
| 4 | entry_index | u32 big-endian; 0-indexed position of this entry within the archive |
| 4 | chunk_index | u32 big-endian; 0-indexed position of this FDAT/SDAT chunk within the entry's data stream |
| 1 | final_flag | `0x01` if this is the final FDAT (or SDAT) chunk before FEND (or SEND); `0x00` otherwise |

Total: **11 bytes**.

Each field's defensive role:
- `Encryption method` + `Cipher mode` — algorithm/mode downgrade detection.
- `entry_index` — intra-archive entry reorder + cross-entry chunk swap detection.
- `chunk_index` — intra-entry chunk reorder detection.
- `final_flag` — truncation detection (a removed last chunk causes the new "last" chunk to authenticate with `final_flag = 0x00` instead of `0x01`).

### 6. Nonce generation

The 12-byte nonce MUST be generated by a cryptographically secure pseudorandom number generator (CSPRNG) for each chunk, per NIST SP 800-38D §8.2.2 (RBG-based construction).

If the CSPRNG fails or blocks (e.g., early-boot Linux without sufficient entropy), the encoder MUST surface the failure and abort archive creation. Silent fallback to a non-cryptographic source is forbidden.

### 7. Key uniqueness budget

NIST SP 800-38D §8.3 limits a single GCM key to 2^32 invocations when nonces are randomly generated.

- **Within a single AEAD entry** under one key: chunk count is bounded by 2^32 (4 billion chunks). At PNA's typical 64 KiB chunk size, this corresponds to 256 TiB per entry — practically unbounded.
- **Across AEAD entries sharing the same PHSF (encoder convention SHOULD)**: total chunk count across all such entries is bounded by 2^32 under one key. At 64 KiB/chunk, 256 TiB total. Encoders writing >2^32 chunks under one password MUST emit at least one entry with a fresh PHSF salt (different key) to reset the counter.

## Security Analysis

### Defenses provided

| Attack | Defense mechanism | Detected? |
|---|---|---|
| Single chunk content forgery | GCM tag | ✅ |
| Single chunk truncation (mid-chunk byte cut) | GCM tag (chunk length authenticated by GCM) | ✅ |
| Intra-entry chunk reorder | AAD `chunk_index` | ✅ |
| Intra-entry truncation (drop last chunk) | AAD `final_flag` | ✅ |
| Cross-entry chunk swap (within same archive, same key) | AAD `entry_index` | ✅ |
| Intra-archive entry reorder | AAD `entry_index` | ✅ |
| Algorithm/mode downgrade (e.g., AES → Camellia switch in FHED) | AAD `Encryption method` + `Cipher mode` bytes | ✅ |

### Defenses NOT provided

| Attack | Status | Notes |
|---|---|---|
| Cross-archive entry substitution (same password, same PHSF salt across distinct archives) | ❌ | Same as PNA Minor 0 (CBC/CTR). Mitigation: encoders MUST regenerate PHSF salt per archive (already required for password hashing security). |
| Archive truncation (drop trailing entries entirely) | ❌ | Same as PNA Minor 0. Would require an authenticated archive-level chunk (AEND with tag). Out of scope for this redesign; can be added in Minor 2 without breaking Minor 1 archives. |
| FHED metadata tampering (filename, permissions, timestamps, xattr) | ❌ (partial) | `Encryption method` and `Cipher mode` bytes ARE authenticated via AAD. Other FHED fields and ancillary chunks (fSIZ, cTIM, mTIM, fPRM, xATR, fLTP) are NOT authenticated. Mitigation: encoders that need this can deploy detached signature schemes outside this spec; a future PNA extension MAY add authenticated-metadata support. |

The two "not detected" attacks are pre-existing gaps in PNA Minor 0. This redesign does not regress relative to Minor 0; it adds 5 new defenses without removing any.

### Why no HKDF

In archive-level AEAD designs, HKDF derives per-entry keys from a single archive master key. That separation matters when nonce generation is deterministic (counter-based): per-entry keys keep each (key, chunk_index) scope independent, so chunk_index can reset per entry without nonce reuse.

In this per-entry AEAD design, nonces are random 96-bit values stored inline. Nonce uniqueness within (key, nonce) scope is maintained by CSPRNG randomness, not by counter resets. The 2^32 invocations cap is shared across all chunks under one key; encoders managing this budget can emit a fresh PHSF salt to start a new key when needed.

Removing HKDF eliminates one cryptographic primitive that community implementations (Java/Go/Python) would otherwise need to ship and maintain.

### Why no magic in AAD

Cross-protocol attacks (where an attacker reuses a leaked PNA key in a different protocol with collision-prone AAD layout) are mitigated by:

1. **PHSF salt is per-archive random**, so the derived 32B key is archive-unique. Attackers leaking the key from one archive cannot reuse it across archives without the password.
2. **Future PNA AEAD framework versions MUST allocate a new FHED `Cipher mode` value** (e.g., a hypothetical PNA AEAD v2 would use `Cipher mode = 3`). The 1-byte `Cipher mode` field in AAD provides intra-PNA version domain separation.
3. **Cross-protocol attacks require an attacker to control both protocols' message formats**, which is out of PNA's threat model.

A 4-byte ASCII magic (e.g., "PAE1") would add 4 B/chunk to the AAD without measurably improving security under PNA's threat model.

   - Remove `§4.1.x AENC Archive ENcryption Context` section (added in `a31aa01`).
   - Remove `§4.1.x FENC File ENcryption Context` section (added in `a31aa01`).
   - Update AHED `Minor version` table entry for Minor 1: change description from "Introduces AENC/FENC chunk types" to "Reserves FHED `Cipher mode` value 2 for GCM (AEAD)".
   - Update FHED Cipher mode table notes (`Cipher mode = 2` → references the per-FDAT layout in §7.5, not AENC/FENC).
   - In PHSF section, add a note: "When the entry uses an AEAD `Cipher mode` (e.g., 2 = GCM), the password hash output is used directly as the GCM key; encoders SHOULD emit byte-identical PHSF chunks across AEAD entries within an archive that share a password."

2. **`cipher_modes/index.md`**:
   - Replace §7.4.2 "Associated Data (AAD)" with the 11-byte AAD construction above.
   - Replace §7.4.3 "Nonce Derivation" with §6 "Nonce generation" above (random CSPRNG, not derived).

   - Remove §8.3 "HKDF (HMAC-based Key Derivation Function)" section entirely.
   - In §8.1 PBKDF2 and §8.2 Argon2 sections, add a note: "When deriving the per-entry GCM key for AEAD entries (FHED `Cipher mode = 2`), the password-hash output of length 32 bytes is used directly as the GCM key; no further key derivation is applied."

   - Update §12.3 "AEAD-specific decoder behavior" to remove all references to AENC/FENC consistency checks.
   - Simplify §12.3.4 "Multipart consistency" — the section becomes a brief note that, since AEAD context is per-entry, multipart archives need no special AEAD-level consistency check beyond the standard part-linking rules.
   - Update §12.3.5 "Error reporting" categories: remove "Cryptographic configuration error: missing AENC / FENC".

5. **`references/index.md`**:
   - Keep NIST SP 800-38D, RFC 6367, RFC 3713, FIPS-197, Bellare-Namprempre 2000, Joux 2006, Vaudenay 2002.
   - **Remove** RFC 5869 (HKDF) — no longer used.

## Implementation impact (libpna)

The libpna AEAD prototype skeleton (commit `c063688f` on `lib/aead-prototype` branch) requires the following adjustments before activation:

- Remove `AeadContext`'s `archive` and `entry` field; AeadContext becomes a thin holder of the per-entry `(key, encryption_method, cipher_mode)` triple plus the chunk-index counter.
- Remove `build_aad` of length 86; replace with a 11-byte builder.
- Remove `build_nonce` derivation logic; replace with `csprng.fill_bytes(&mut nonce_buf)`.
- Remove `aead/` references to HKDF.
- Skip `AENC` / `FENC` chunk type definitions in `lib/src/chunk/types.rs`.
- The dependency list shrinks: `hkdf = "0.13"` is no longer needed.

## Out-of-scope (deferred)

- **Authenticated archive-level metadata** (defense against archive truncation, FHED metadata tampering): can be added in a future PNA Minor 2 via a new chunk type without breaking Minor 1 archives.
- **Streaming AEAD with chunk-counter nonce derivation** (saves 12 B per chunk by deriving nonce instead of inlining): can be added as a separate Cipher mode value (e.g., `Cipher mode = 3` for "GCM-deterministic-nonce") in a future Minor.
- **External cryptographic audit**: vendor selection and external review of this redesigned spec before declaring it stable.
